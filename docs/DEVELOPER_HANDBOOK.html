🚀 Quick Start Guide

Welcome to the project! The AI Personal Teacher System is built to be modular, secure, and production-ready. The complete containerized application stack can be built and launched locally using a single configuration command.

⚠️ Networking Alert: The development Docker Compose stack binds to host port 8081. This ensures the environment runs cleanly without conflicting with default web services or local Nginx configurations utilizing ports 80 or 8080.
1

Checkout codebase and instantiate environments

Clone the project files, navigate to the project root directory, and duplicate the environment config templates:

# Copy template variables
cp .env.example .env
cp backend/.env.example backend/.env
cp frontend/.env.example frontend/.env
2

Initialize and start docker stack

Trigger the Docker Compose execution. This pulls base images, executes multi-stage builds for components, and starts background services:

docker compose up --build
3

Verify application container health

Confirm that the system services are accepting traffic by launching the following verification routes:

Component Access URL Target Protocol Purpose
Nginx Proxy Entrypoint http://localhost:8081 HTTP / HTTPS Gateway proxy routing to Web & API services
Vite Frontend http://localhost:8081/ HTTP + WSS (HMR WebSocket) React landing dashboard
FastAPI Docs http://localhost:8081/docs HTTP Swagger specification utility
API Health Check http://localhost:8081/api/v1/health JSON HTTP FastAPI Verification check for database and broker

🔄 System Data Flow & Request Lifecycle

Understanding the request path is critical for debugging connection state issues. Below is a trace of the network data flow through the architecture:

[Client Web Browser]
         │
         │ HTTP / WS (Port 8081)
         ▼
 ┌───────────────┐          /api/v1/*          ┌───────────────┐
 │  leen-nginx   │────────────────────────────▶│   leen-api    │
 │ (Proxy Gate)  │                             │  (FastAPI)    │
 └───────────────┘                             └───────┬───────┘
         │                                             │
         │ / (Static React Assets)                     │ Write/Read DB
         ▼                                             ▼
 ┌───────────────┐                             ┌───────────────┐
 │   leen-web    │                             │ leen-postgres │
 │ (Vite Server) │                             │   (Postgres)  │
 └───────────────┘                             └───────────────┘
                                                       │
                                                       │ Trigger Task
                                                       ▼
 ┌───────────────┐     Fetch Task Payload      ┌───────────────┐
 │  leen-worker  │◀────────────────────────────│  leen-redis   │
 │   (Celery)    │                             │  (Redis MQ)   │
 └───────────────┘                             └───────────────┘

Lifecycle Steps

  1. Proxy Ingress: The user accesses http://localhost:8081. The request hits leen-nginx.
  2. Static Assets: In development, requests for pages and files are proxied to the Vite server (`leen-web`). In production, Nginx serves pre-built React files directly from the static volume.
  3. API Requests: Requests prefixed with /api/v1/ are forwarded to leen-api (FastAPI running on port 8000 internally).
  4. Data Resolution: The API reads or writes to leen-postgres. If a request demands long-running execution (e.g. email dispatching, AI generation), it schedules a Celery task.
  5. Task Brokerage: The task metadata is serialized to JSON and pushed to `leen-redis` Database 0 (the Celery message queue).
  6. Async Processing: The Celery worker (`leen-worker`) picks up the task asynchronously, processes it, and writes the status and return data back to Redis Database 1 (results backend).

🌿 Git & Branching Workflow

To preserve branch integrity and guarantee a stable staging environment, the team follows strict branching rules. Direct commits to the stable branch (main) are blocked. Changes are reviewed through pull requests.

Workflow Standard Development Flow

1. Pull down the latest main branch state before beginning work.

2. Create a clean branch from main using naming rules.

3. Format and check lint rules before pushing.

4. Push and create a PR targeting the main branch.

Review Peer Validation

Every pull request demands a minimum of 1 code approval from another team member.

All automated checks (linters, backend tests) must pass successfully before merging.

Git Branch Naming Conventions

Branch Type Name Format Example

Bug fixes bugfix/<issue-number>-<hyphenated-desc> bugfix/45-fix-db-timeout-leak
Documentation updates docs/<hyphenated-desc> docs/update-architecture-manual

We use the Conventional Commits standard. This ensures git logs are clear and enables automated release notes.

[Optional longer body detailing breaking changes or ticket references]

Commit Types Checklist

  • feat: Incorporating new capabilities (e.g. feat(api): expose register endpoints).
  • fix: Repairing logic defects (e.g. fix(frontend): adjust register state reset).
  • refactor: Improving design quality without code changes (e.g. refactor(database): extract session pooler).
  • docs: Modifying markdown, instructions, documentation (e.g. docs(handbook): add rebase guides).
  • test: Building or editing checking suites (e.g. test(api): write login endpoints validations).
  • chore: Changing dependencies, build tasks, configurations (e.g. chore(deps): upgrade fastapi).

🛡️ Conflict Resolution & Event Management

In a team environment, merge conflicts on configuration files or database schemas can occur. Follow these procedures to resolve conflicts cleanly.

⚠️ Alembic Database Migration Alert: Never edit Alembic migration scripts manually. If developers merge features that generate parallel database migration states, it creates split head branches.

Resolving Migration Splits

If Alembic reports a Multiple Heads Detected database error, run the following commands to create a merge revision:

# 1. Inspect migration history branch heads
alembic heads

# 2. Create a merge revision linking both heads
alembic merge -m "merge branch heads" <head-id-1> <head-id-2>

# 3. Apply changes to local PostgreSQL container
alembic upgrade head

Resolving Standard Git Conflicts

Always fetch and rebase your branch against the latest target branch to verify your conflicts locally before submitting PRs:

# 1. Pull changes
git checkout main
git pull origin main

# 2. Start rebase on your branch
git checkout feature/your-feature-branch
git rebase main

# 3. Resolve conflicts highlighted in files, then stage:
git add <resolved-files>

# 4. Continue rebase
git rebase --continue

🐍 Backend Coding Standards (Python/FastAPI)

We use Python 3.12 features and require code structure to compile cleanly. Below are our core coding guidelines.

1. Type Safety with Type Hints

Type hints are mandatory for all function parameters, class attributes, and return signatures. Use standard collections types (e.g. list[str] instead of List[str]) and the pipe operator for optional variables:

def fetch_user_profile(user_uuid: str, db_session: Session) -> User | None:
    return db_session.query(User).filter(User.id == user_uuid).first()

2. Dependency Injection

FastAPI uses dependency injection via Depends. Use dependencies to resolve database sessions, verify authentication tokens, and fetch caching clients. This decouples logic from routing:

from fastapi import APIRouter, Depends
from sqlalchemy.orm import Session
from app.database import get_db

router = APIRouter()

@router.get("/users/me")
def read_current_user(db: Session = Depends(get_db)):
    # The db session is resolved, used, and closed automatically
    return {"message": "Success"}

3. Secure Handling of Sensitive Data

Never log sensitive values like user passwords, authorization tokens, or environment keys. Passwords must be hashed using bcrypt or Argon2 prior to storage. Do not store tokens in the database as plaintext; always hash session keys or tokens at rest.

⚛️ Frontend Coding Standards (React/JavaScript)

The frontend is a lightweight, responsive React 19 application built using Vite. We use JavaScript (ES6+) and TailwindCSS v4.

1. Component Architecture

Components must be functional declarations. Avoid legacy class-based architectures. Group code logic, states, and event handles cleanly before the return block:

// PascalCase file naming: UserCard.jsx
import React from 'react';

export function UserCard({ username, userRole }) {
  const isAdmin = userRole === 'admin';

  return (
    <div className="glass-card p-6 border border-white/10 rounded-xl">
      <h4 className="text-white text-lg font-bold">{username}</h4>
      <span className={`badge ${isAdmin ? 'badge-success' : 'badge-warning'}`}>
        {userRole}
      </span>
    </div>
  );
}

2. React Query for Server-State Caching

Avoid caching API responses in global React state (like Redux or Context) when they can be managed by **React Query**. React Query handles cache invalidations and fetches: 

import { useQuery } from '@tanstack/react-query';
import axios from 'axios';

function useFetchHealth() {
  return useQuery({
    queryKey: ['api-health'],
    queryFn: async () => {
      const response = await axios.get('/api/v1/health');
      return response.data;
    },
    staleTime: 60000, // Keep cache fresh for 60 seconds
  });
}

🔌 API Documentation & Schema Design

Well-documented APIs make client integration easier. Use the following standards to document your code for Swagger/OpenAPI generators:

  • Response Models: Always specify the response_model parameter in FastAPI decorators. This ensures response payload structures are checked and serialized correctly.
  • Schema fields: Use Pydantic's Field to document parameter rules and describe what each key contains.
  • Route summaries: Write clean, descriptive docstrings inside route handlers. The first line forms the route summary, and subsequent lines populate the endpoint description in Swagger.

API Endpoint Implementation Blueprint

from fastapi import APIRouter, HTTPException, status
from pydantic import BaseModel, Field

router = APIRouter()

    email: str = Field(..., description="Registered user email", example="team@leen.ai")
    is_active: bool = Field(True, description="Account status flag")

@router.get(
    "/users/{user_id}",
    response_model=UserDetailsResponse,
    status_code=status.HTTP_200_OK,
    summary="Retrieve details of a user",
    response_description="User details payload"

def fetch_user_details(user_id: str):
    """
    Look up user record in the relational store.
    
    If the user record is missing, returns a 404 error response.
    """
    user_record = find_user_by_id(user_id)
    if not user_record:
        raise HTTPException(

💬 Commenting & Documentation Guidelines

Write comments to explain the **context** and **decisions** behind the code. Do not restate what the code does step-by-step.

Good Comment Context

Explain why a specific third-party client configuration or timeout parameter was chosen.

Document technical choices made to address performance bottlenecks or safety concerns.

Bad Redundant Comments

Avoid simple, line-by-line descriptions (e.g., # Increment x by 1).

Do not leave dead or commented-out code blocks. Rely on Git history for changes.

🔧 Troubleshooting FAQ

Q: Celery worker reports connection timeouts to Redis.

A: Ensure the Redis service is healthy. Check container health status with docker compose ps. If necessary, stop and restart the containers with docker compose down && docker compose up.

Q: Changes in Python backend do not show up in container.

A: Verify that the volume mount is active. In development mode, the local directory is mounted to /app inside the container, which allows Uvicorn and Celery to reload code automatically on save.
